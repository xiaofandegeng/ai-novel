# AI 小说创作工作台开发顺序

版本：v0.1
日期：2026-04-28
用途：交给其他 AI 开发代理时使用的执行顺序说明

## 0. 开发代理必须先阅读的文档

在写任何代码前，AI 开发代理必须先阅读：

1. 产品设计文档：`docs/product/ai-novel-workbench-product-design.md`
2. UI 设计规格：`docs/design/ai-novel-workbench-ui-design-spec.md`
3. 当前修复文档：`docs/development/remediation-plan-2026-04-28.md`

当前状态提示：如果修复文档存在且其中的验收命令尚未全部通过，必须先完成修复收敛阶段，不得继续新增后续阶段功能。

开发原则：

1. 先搭基础，再做页面。
2. 先做可闭环的 MVP，再扩展高级功能。
3. 所有 UI 必须遵守 UI 设计规格中的 token、布局、组件和状态规则。
4. AI 生成结果必须先进入确认区，不得直接覆盖用户正文。
5. 每个阶段完成后必须能本地运行、能验证、能继续扩展。

## 1. 总体开发顺序

```text
阶段 0：项目初始化
  ↓
阶段 1：设计系统与基础 UI
  ↓
阶段 2：数据模型与本地基础服务
  ↓
阶段 3：项目中心与故事圣经
  ↓
阶段 4：人物管理
  ↓
阶段 5：大纲规划
  ↓
阶段 6：章节写作闭环
  ↓
阶段 7：AI 调用抽象层
  ↓
阶段 8：人物关系、冲突、伏笔
  ↓
阶段 9：知识库上传与拆书
  ↓
阶段 10：质量评估
  ↓
阶段 11：版本历史、写作计划、风格画像
  ↓
阶段 12：响应式、可访问性、最终打磨
```

## 阶段 0：项目初始化

目标：建立可运行的工程骨架。

任务：

1. 初始化前端项目。
2. 初始化后端项目。
3. 配置 TypeScript。
4. 配置代码格式化和 lint。
5. 配置环境变量。
6. 配置本地数据库连接。
7. 建立基础目录结构。

建议目录：

```text
apps/web
apps/api
packages/shared
packages/ui
docs
```

验收：

1. 前端可本地启动。
2. 后端可本地启动。
3. 前后端健康检查正常。
4. README 记录启动命令。

## 阶段 1：设计系统与基础 UI

目标：先把 UI 基础定下来，避免后续页面风格混乱。

任务：

1. 建立颜色 token。
2. 建立字体、字号、间距、圆角、阴影 token。
3. 封装基础布局组件。
4. 封装 Button、IconButton、Input、TextArea、Select、Tag。
5. 封装 Panel、Drawer、Modal、Toast。
6. 封装 EmptyState、LoadingState、ErrorState。
7. 引入 Lucide 图标。

禁止：

1. 禁止在业务页面中直接写裸色值。
2. 禁止先做复杂页面再补设计系统。
3. 禁止使用 emoji 作为功能图标。

验收：

1. 有统一 token 文件。
2. 基础组件可复用。
3. 组件具备 hover、active、focus、disabled、loading 状态。
4. icon-only 按钮具备 aria-label 和 tooltip。

## 阶段 2：数据模型与本地基础服务

目标：建立 MVP 所需的数据结构。

优先数据模型：

1. NovelProject
2. StoryBible
3. Character
4. Volume
5. Chapter

任务：

1. 建立数据库 schema。
2. 建立基础 CRUD。
3. 建立统一 API 响应格式。
4. 建立错误处理。
5. 建立基础 seed 数据。

验收：

1. 能创建小说项目。
2. 能创建故事圣经。
3. 能创建人物。
4. 能创建分卷和章节。
5. 前端能读取并展示 seed 数据。

## 阶段 3：项目中心与故事圣经

目标：完成小说项目的入口和顶层设定管理。

页面：

1. 项目列表页。
2. 项目首页。
3. 故事圣经页。

任务：

1. 创建项目。
2. 编辑项目基础信息。
3. 展示项目状态。
4. 编辑故事主题、世界观、主线冲突、关键规则、时间线。
5. 保存故事圣经。

验收：

1. 用户能创建一个小说项目。
2. 用户能进入项目工作台。
3. 用户能编辑并保存故事圣经。
4. 页面符合三栏工作台风格。
5. 所有表单有 loading、error、empty 状态。

## 阶段 4：人物管理

目标：维护人物档案，为后续大纲和正文生成提供上下文。

页面：

1. 人物管理页。

任务：

1. 创建人物。
2. 编辑人物基础资料。
3. 维护目标、恐惧、秘密、欲望、弱点、性格、成长弧线。
4. 展示人物列表和人物详情。
5. 支持按角色类型筛选。

验收：

1. 能创建主角、配角、反派等人物。
2. 人物详情保存后刷新不丢失。
3. 人物列表支持搜索或筛选。
4. 人物详情页符合 UI 设计规格。

## 阶段 5：大纲规划

目标：实现全书、分卷、章节三级大纲。

页面：

1. 大纲规划页。

任务：

1. 创建分卷。
2. 创建章节。
3. 编辑章节标题和摘要。
4. 编辑章节详细大纲。
5. 建立章节树。
6. 支持章节状态：未开始、规划中、写作中、已完成。

验收：

1. 能创建至少 1 个分卷和 10 个章节。
2. 能编辑单章大纲。
3. 章节树状态清晰。
4. 章节大纲字段与产品设计文档一致。

## 阶段 6：章节写作闭环

目标：实现最小写作闭环，这是 MVP 的核心。

页面：

1. 章节写作页。

任务：

1. 实现章节树。
2. 实现正文编辑器。
3. 实现自动保存或手动保存。
4. 实现字数统计。
5. 实现右侧上下文面板。
6. 展示当前章节大纲、相关人物、前文摘要。

验收：

1. 用户能选择章节。
2. 用户能编辑正文。
3. 正文能保存。
4. 刷新后正文不丢失。
5. 页面中正文编辑器是视觉中心。
6. 保存状态清楚显示：未保存、保存中、已保存、保存失败。

## 阶段 7：AI 调用抽象层

目标：让 AI 能力接入系统，但不和页面强耦合。

任务：

1. 建立 LLM Provider 接口。
2. 支持结构化生成。
3. 支持普通文本生成。
4. 支持失败重试。
5. 支持上下文组装。
6. 建立 AI 操作记录。

优先 AI 功能：

1. 根据一句话创意生成故事圣经。
2. 根据故事圣经生成人物。
3. 生成全书大纲。
4. 生成章节大纲。
5. 根据章节大纲生成正文。
6. 总结章节。

关键规则：

1. AI 结果必须进入确认面板。
2. 用户确认后才能插入、替换或保存。
3. AI 请求失败必须显示可理解错误。

验收：

1. 能通过统一接口调用 AI。
2. 至少完成“生成章节正文”能力。
3. AI 结果不会自动覆盖用户正文。
4. 页面展示 AI 使用的上下文来源。

## 阶段 8：人物关系、冲突、伏笔

目标：开始实现长篇小说记忆管理。

新增数据模型：

1. Relationship
2. Conflict
3. Foreshadowing

页面：

1. 人物关系页。
2. 冲突矩阵页。
3. 伏笔管理页。

任务：

1. 创建人物关系。
2. 创建冲突。
3. 创建伏笔。
4. 支持状态管理。
5. 在章节写作页右侧上下文面板展示相关关系、冲突和伏笔。

验收：

1. 用户能维护人物关系。
2. 用户能维护冲突状态。
3. 用户能维护伏笔状态。
4. 写作时能看到相关上下文提醒。

## 阶段 9：知识库上传与拆书

目标：支持上传参考作品并形成可检索知识库。

新增数据模型：

1. KnowledgeSource
2. KnowledgeChunk
3. WritingTechnique

页面：

1. 知识库页。

任务：

1. 支持 TXT 上传。
2. 提取文本。
3. 自动拆分章节。
4. 生成章节摘要。
5. 生成全书摘要。
6. 抽取写作技巧。
7. 入库保存。

后续扩展：

1. Markdown。
2. EPUB。
3. PDF。
4. 向量检索。

验收：

1. 用户能上传 TXT。
2. 系统能拆分章节。
3. 系统能展示摘要。
4. 知识库页面展示上传、解析、失败状态。
5. 页面有版权提示。

## 阶段 10：质量评估

目标：让系统从“生成内容”升级到“辅助改稿”。

新增数据模型：

1. QualityReport

页面：

1. 质量报告页。

任务：

1. 对单章生成质量报告。
2. 对全书生成阶段性质量报告。
3. 展示评分维度。
4. 展示问题列表。
5. 展示修改建议。
6. 支持根据建议触发局部改写。

验收：

1. 能生成单章质量报告。
2. 报告能定位到章节。
3. 问题按严重程度排序。
4. 修改建议可执行。

## 阶段 11：版本历史、写作计划、风格画像

目标：提升长期创作体验。

任务：

1. 保存章节历史版本。
2. 对比版本差异。
3. 恢复历史版本。
4. 设置每日目标字数。
5. 展示写作进度。
6. 根据作者正文提取风格画像。

验收：

1. 用户能查看章节历史版本。
2. 用户能恢复旧版本。
3. 用户能设置写作目标。
4. 系统能展示基础写作统计。

## 阶段 12：响应式、可访问性、最终打磨

目标：让产品达到可长期使用的质量。

任务：

1. 完成平板端适配。
2. 完成移动端适配。
3. 检查无横向滚动。
4. 检查键盘导航。
5. 检查 aria-label。
6. 检查焦点状态。
7. 检查 loading、empty、error 状态。
8. 检查文本溢出。
9. 检查 AI 结果确认流程。

验收：

1. 桌面端、平板端、移动端均可使用。
2. 所有 icon-only 按钮有 aria-label。
3. 所有表单有 label 和错误提示。
4. 页面无明显视觉错位。
5. 章节写作页适合长时间写作。

## 13. 给 AI 开发代理的工作规则

### 13.1 每次开发前

AI 开发代理应先说明：

1. 当前执行哪个阶段。
2. 会修改哪些文件。
3. 完成后如何验证。

### 13.2 每次开发中

AI 开发代理应遵守：

1. 不跳阶段。
2. 不重写无关代码。
3. 不引入未经确认的大型依赖。
4. 不把 UI 写成临时样式。
5. 不绕过设计 token。

### 13.3 每次开发后

AI 开发代理必须输出：

1. 修改了哪些文件。
2. 完成了哪些功能。
3. 如何运行。
4. 如何测试。
5. 是否有未完成事项。

如果要马上开始开发，建议第一批任务这样拆：

输入：

1. 产品设计文档。
2. UI 设计规格。
3. 本开发顺序文档。

输出：

1. 可运行的前端项目。
2. 可运行的后端项目。
3. 基础目录结构。
4. README 启动说明。

### 任务 B：建立设计系统

输入：

1. UI 设计规格第 4 到第 9 节。

输出：

1. 设计 token。
2. 基础组件。
3. 基础布局。

### 任务 C：建立 MVP 数据模型

输入：

1. 产品设计文档阶段 1。
2. 本文阶段 2。

输出：

1. 数据库 schema。
2. 基础 CRUD。
3. seed 数据。

### 任务 D：实现项目中心和故事圣经

输入：

1. 产品设计文档阶段 1。
2. UI 设计规格 10.1、10.2、10.3。

输出：

1. 项目列表页。
2. 项目首页。
3. 故事圣经页。

## 15. 最小可交付版本定义

最小可交付版本必须包含：

1. 项目创建。
2. 故事圣经编辑。
3. 人物管理。
4. 大纲规划。
5. 章节写作。
6. 章节正文保存。
7. 一个可用的 AI 生成入口。
8. AI 结果确认面板。
9. 基础 UI 设计系统。

不满足以上内容，不应认为 MVP 完成。
