# AI Collaboration Rules

版本：v0.1
日期：2026-04-29
用途：约束 AI 写作能力、AI 开发代理行为和用户内容安全边界。

## 1. 产品内 AI 的基本边界

AI 小说创作工作台中的 AI 是作者助手，不是自动接管作者作品的系统。

核心规则：

1. AI 可以建议、生成、总结、评分、改写。
2. AI 不能在用户未确认时覆盖正文、大纲、设定、人物、关系、冲突。
3. AI 结果必须可预览、可选择去向、可丢弃。
4. 用户必须能知道 AI 使用了哪些上下文。
5. 任何会造成数据覆盖的操作必须二次确认。

## 2. AI 结果确认区

所有 AI 生成结果先进入 pending result。

标准动作：

- 插入：追加到光标位置或指定字段末尾。
- 替换：覆盖指定字段，必须说明覆盖范围。
- 保存为备选：保留为候选结果，不改变当前正文。
- 丢弃：删除候选结果。

禁止：

```text
AI response -> directly assign to draft
AI response -> directly assign to outline
AI response -> directly update character/conflict/relationship
```

## 3. 写作页 AI 规则

写作页是最高风险页面。

规则：

1. AI 扩写、续写、润色、改写结果必须进入确认区。
2. 替换正文前必须显示覆盖风险。
3. 自动保存不得把未确认 AI 内容写入章节草稿。
4. AI 面板必须区分“生成中”“可应用”“已应用”“已丢弃”。
5. 版本快照应在高风险替换前后可用。

## 4. 大纲页 AI 规则

规则：

1. AI 大纲建议不能直接覆盖章节大纲。
2. 生成分卷、章节、场景时先展示候选结构。
3. 批量创建章节前必须展示将创建的条目数量。
4. 用户确认后再写入数据库。

## 5. 知识库 AI 规则

知识库支持上传经典小说或参考作品，但必须避免仿写和版权风险。

规则：

1. 知识库目标是结构分析、技巧总结、节奏拆解，不是逐段仿写。
2. 上传文本要拆分 source、chunk、note 等结构。
3. AI 总结应提炼技巧、结构、人物关系、冲突推进方式。
4. 后续写作引用知识库时，优先给出抽象技巧和结构建议。
5. 不生成与参考作品高度相似的连续表达。

## 6. 质量评估 AI 规则

质量评估应产生结构化报告：

- 总分
- 节奏评分
- 冲突评分
- 逻辑评分
- 人物一致性评分
- 风格评分
- 问题列表
- 修改建议

规则：

1. 评分结果必须保存成报告，方便版本对比。
2. 建议不能直接改正文。
3. “应用建议”必须进入写作页确认区。

## 7. 开发代理行为规则

AI 开发代理必须：

1. 先读 `AGENTS.md` 和相关 development 文档。
2. 先修 failing checks，再新增功能。
3. 保持每次修改可审查。
4. 对数据库、AI 覆盖、删除恢复类操作保持保守。
5. 在最终回复中说明验证命令和结果。

AI 开发代理不得：

1. 因为页面能打开就声称阶段完成。
2. 忽略 `pnpm build`、`pnpm lint`、`pnpm test`。
3. 为了通过类型检查使用大面积 `any`。
4. 绕过设计系统写临时 UI。
5. 把 mock 数据伪装成完整功能。

## 8. 阶段验收口径

一个功能阶段只有同时满足以下条件才算完成：

1. UI 入口存在。
2. API 流程存在。
3. 数据能持久化。
4. loading、error、empty 状态存在。
5. AI 或危险操作有确认边界。
6. `pnpm check` 通过。
7. 文档或 README 中记录了使用方式。
