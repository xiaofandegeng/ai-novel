# 下一步正式写作可用性验收与维护文档

日期：2026-05-14
状态：待实施
前置假设：

1. `docs/development/next-step-production-readiness-plan-2026-05-14.md` 已完成。
2. 数据完整性、导入导出、AI 错误恢复、性能优化、核心自动化验收已完成。
3. 项目已经具备从设定、大纲、写作、章后分析、记忆回流到健康监控的完整能力。

目标：在正式长期写作前，完成一次“真实作者视角”的最终验收，建立后续维护、备份、回归和迭代机制。

---

## 1. 本轮实施范围

本轮不再新增大型功能，只做正式写作前的可用性确认和维护体系：

1. 真实小说试写验收。
2. 全模块回归矩阵。
3. AI 生成质量验收。
4. 长篇连续性验收。
5. 数据备份和恢复演练。
6. 问题分级和修复 SLA。
7. 后续迭代入口文档。

暂不做：

- 新 AI provider。
- 新 UI 大改版。
- 新故事结构系统。
- 多用户协作。
- 云部署。

---

## 2. 阶段 A：真实小说试写验收

### 2.1 目标

用真实写作流程验证系统，而不是只验证按钮和接口。

### 2.2 测试项目

使用固定测试小说：

```text
项目名：镜中城
类型：都市奇幻 / 记忆悬疑
主题：身份、记忆与选择的代价
目标字数：200000
```

核心角色：

```text
林澈：主角，能读取他人记忆碎片。
沈微：盟友，档案馆调查员，隐瞒真实身份。
楚笙：反派，经营记忆交易网络。
游箴：小角色，记忆黑市跑腿人。
```

### 2.3 试写范围

至少完成：

1. 1 个完整故事设定集。
2. 4 个角色。
3. 4 条人物关系。
4. 3 条核心矛盾。
5. 4 条伏笔。
6. 2 卷大纲。
7. 6 章章节规划。
8. 3 章正文草稿。
9. 3 次章后分析。
10. 1 次导出导入恢复。

### 2.4 验收流程

```text
创建项目
  ↓
配置 AI provider
  ↓
填写故事设定
  ↓
AI 辅助补充设定
  ↓
生成 / 补充角色
  ↓
自动生成关系
  ↓
配置矛盾和伏笔
  ↓
规划大纲和章节元素
  ↓
启动写作任务
  ↓
确认计划
  ↓
确认草稿写入
  ↓
章后分析
  ↓
接受结构化建议
  ↓
查看健康指标
  ↓
导出项目
  ↓
导入项目
  ↓
继续写下一章
```

### 2.5 通过标准

- [ ] 作者无需手动复制 AI 建议到字段。
- [ ] AI 结果均先进入确认区。
- [ ] 关系、矛盾、伏笔、事实、记忆能自动回流。
- [ ] 下一章 AI 能读取上一章沉淀数据。
- [ ] 导出导入后能继续写作。
- [ ] 健康页能给出可解释风险。

## 3. 阶段 B：全模块回归矩阵

### 3.1 目标

明确每次大改后必须检查哪些模块，防止一个闭环修好了另一个闭环断掉。

### 3.2 回归矩阵

| 模块 | 必测操作 | 通过标准 |
| --- | --- | --- |
| 项目列表 | 新建、搜索、删除、打开 | 不重叠，不误删，有确认 |
| 项目设置 | 基础信息、AI 配置、人格绑定 | 不覆盖未加载配置，检测可用 |
| 故事设定集 | AI 补充、确认、保存 | AI 输出清洗后进入字段 |
| 角色管理 | AI 分析、AI 新增角色、批量小角色 | 字段回填，关系自动候选 |
| 人物关系 | 新建、去重、AI 回填 | 不重复，跨项目不可写 |
| 矛盾矩阵 | 绑定角色、章后更新 | 结构化参与者生效 |
| 伏笔台账 | 绑定角色、兑现、章后更新 | 状态真实变化 |
| 知识库 | 上传、分析、RAG 召回 | 不把原文注入 prompt |
| 大纲规划 | 角色、章节元素、场景 | 写作页上下文一致 |
| 正文写作 | 任务流、确认写入、版本 | 不直接覆盖正文 |
| 章后分析 | 接受、拒绝、应用 | 业务表真实更新 |
| 健康指标 | 刷新、解释、跳转 | 风险有证据 |
| 导入导出 | 导出、删除、导入 | 数据完整恢复 |
| AI 上下文调试 | 查看快照 | 可追溯，无敏感信息 |

### 3.3 回归频率

1. 每次 schema 修改后跑全量回归。
2. 每次 AI 上下文修改后跑写作闭环回归。
3. 每次 UI 主流程修改后跑浏览器回归。
4. 每次导入导出修改后跑备份恢复回归。

## 4. 阶段 C：AI 生成质量验收

### 4.1 目标

确认 AI 不是“能返回文本”就算成功，而是能按照设定、人物、关系、矛盾、伏笔和写作人格持续产出。

### 4.2 质量维度

每次试写至少人工检查：

1. 是否符合世界观规则。
2. 是否符合角色目标、恐惧、秘密、欲望。
3. 是否沿用当前人物关系。
4. 是否推进核心矛盾。
5. 是否照顾开放伏笔。
6. 是否延续前文记忆。
7. 是否符合绑定人格的风格约束。
8. 是否避免参考作品原文仿写。

### 4.3 评分表

| 维度 | 分值 | 说明 |
| --- | --- | --- |
| 设定一致性 | 1-5 | 是否违反故事设定 |
| 人物一致性 | 1-5 | 是否 OOC |
| 情节连续性 | 1-5 | 是否接上前文 |
| 冲突推进 | 1-5 | 是否推动核心矛盾 |
| 伏笔管理 | 1-5 | 是否新增或回应伏笔 |
| 风格一致性 | 1-5 | 是否符合写作人格 |
| 可读性 | 1-5 | 是否像可继续修改的正文 |

通过标准：

```text
平均分 >= 4
且任一维度不得低于 3
```

### 4.4 不通过处理

如果不通过：

1. 查看 AI 上下文快照。
2. 判断是上下文缺失、prompt 约束弱、模型质量差，还是数据未回流。
3. 记录到 `docs/testing/authoring-acceptance-report.md`。
4. 修复后重新生成同一章。

## 5. 阶段 D：长篇连续性验收

### 5.1 目标

验证系统能支持多章连续写作，而不是只适合单章生成。

### 5.2 最小连续性测试

连续完成 3 章：

```text
第 1 章：林澈发现记忆异常。
第 2 章：沈微介入调查。
第 3 章：楚笙的记忆交易网络浮出水面。
```

每章都必须：

1. 启动写作任务。
2. 确认计划。
3. 生成草稿。
4. 一致性检查。
5. 确认写入。
6. 章后分析。
7. 接受至少一条结构化建议。
8. 刷新健康指标。

### 5.3 连续性检查项

- [ ] 第 2 章能引用第 1 章记忆。
- [ ] 第 3 章能引用第 1-2 章关系变化。
- [ ] 伏笔不会无故消失。
- [ ] 人物关系不会回退到旧状态。
- [ ] 矛盾强度能持续变化。
- [ ] 健康指标能发现未推进风险。

---

## 6. 阶段 E：备份与恢复演练

### 6.1 目标

确认用户长期写作时不会因为误操作、迁移或 AI 处理失败丢数据。

### 6.2 演练流程

1. 完成 3 章测试写作。
2. 导出完整项目备份。
3. 记录导出报告中的数据数量。
4. 删除项目。
5. 导入项目。
6. 对比数据数量。
7. 打开第 4 章继续写作。
8. 确认 AI 上下文能读取导入后的前 3 章数据。

### 6.3 必须对比的数据

```text
characters
character_relationships
conflicts
foreshadowing_items
chapters
chapter_scenes
chapter_elements
chapter_memories
story_fact_triples
knowledge_sources
knowledge_chunks
persona_memory_cards
chapter_versions
health_risks
ai_context_snapshots
```

### 6.4 通过标准

- [ ] 数据数量一致，或有明确可接受差异说明。
- [ ] 导入后项目可打开。
- [ ] 导入后可继续写作。
- [ ] 导入后 AI 上下文可构建。
- [ ] 导入后不会出现跨项目旧 ID。

## 7. 阶段 F：问题分级与修复 SLA

### 7.1 P0 阻断

定义：

- `pnpm check` 失败。
- 数据库迁移失败。
- 项目无法打开。
- 正文写入丢失。
- 导入导出导致数据丢失。
- AI 错误文本可被应用到正文。

处理：

```text
必须立即修复，不允许继续新增功能。
```

### 7.2 P1 关键流程问题

定义：

- 章后分析不能回填。
- 人物关系重复或错链。
- AI 上下文缺少关键设定。
- 写作任务无法继续。
- 健康指标明显错误。

处理：

```text
进入下一轮开发前必须修复。
```

### 7.3 P2 体验问题

定义：

- 文案不统一。
- 操作路径绕。
- 页面信息密度不佳。
- 错误提示不够清晰。

处理：

```text
合并到 UI/UX 批量优化。
```

## 8. 阶段 G：后续迭代入口

正式写作验收通过后，后续新需求必须从以下入口进入：

1. `docs/development/feature-request-template.md`
2. `docs/development/bug-report-template.md`
3. `docs/development/ai-output-quality-report-template.md`
4. `docs/testing/authoring-acceptance-report.md`

### 8.1 feature request 必填

- 解决什么写作问题。
- 影响哪些模块。
- 是否改变 AI 上下文。
- 是否新增数据库字段。
- 是否影响导入导出。
- 验收流程。

### 8.2 bug report 必填

- 复现步骤。
- 期望结果。
- 实际结果。
- 影响范围。
- 是否数据损坏。
- 是否有截图或日志。

### 8.3 AI output quality report 必填

- 使用场景。
- 使用模型。
- 使用章节。
- 上下文快照 ID。
- 输出问题。
- 人工评分。
- 是否可通过 prompt / 上下文 / 数据修复。
